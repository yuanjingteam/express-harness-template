# AI Review

在 agent-check.sh 全部通过后、提交给人工 Review 之前，执行此 skill 进行语义级审查。

## 触发条件

- agent-check.sh 已通过（ESLint + TypeScript + Jest 全绿）
- 即将提交变更给人类 review 之前
- 禁止在 agent-check.sh 未通过时调用此 skill

## 审查流程

### Step 1: 收集变更上下文

使用 `git diff` 获取本次变更的完整 diff（包括未暂存和已暂存的变更）。

重点关注：
- 新增/修改/删除了哪些文件
- 每个变更文件的 diff 内容
- 变更涉及的分层位置（Controller / Service / Repository / DTO / 其他）

### Step 2: 逐项审查

按以下三个维度逐项检查，每项必须给出明确结论：

#### 2.1 过度设计检查

| 检查项 | 判定标准 |
|--------|----------|
| 抽象层 | 是否引入了当前只有一个实现者的接口或抽象类？如果只有一个实现者且可预见的未来也只有一个，则属于过度设计 |
| 配置化 | 是否将硬编码就能解决的简单逻辑提取成了配置文件或环境变量？ |
| 泛化 | 是否为只有一种场景的需求编写了多场景扩展点（如策略模式、插件机制）？ |
| 依赖 | 是否引入了项目当前不需要的 npm 包？ |

#### 2.2 误解需求检查

| 检查项 | 判定标准 |
|--------|----------|
| API 契约 | 变更是否与 `docs/ai/api-contracts.md` 中定义的请求/响应结构一致？ |
| Fixture | 变更是否与 `test/fixtures/` 中 fixture 描述的预期行为一致？ |
| 业务意图 | 代码实现是否真正解决了需求，还是只解决了表面问题？ |

验证方法：将变更与 `docs/ai/api-contracts.md` 和相关 fixture 逐项对照。

#### 2.3 测试自嗨检查

| 检查项 | 判定标准 |
|--------|----------|
| Mock 滥用 | 是否 mock 了所有依赖，导致测试只能验证 mock 本身而非实际行为？ |
| 无断言 | 测试是否只验证了"不报错"而非"结果正确"？ |
| 实现耦合 | 测试是否依赖了被测代码的内部实现细节（如私有方法、内部状态），而非公开的输入/输出？ |
| Fixture 绕过 | 是否存在修改 fixture 以让测试通过的行为？ |

### Step 3: 输出审查报告

按以下格式输出报告：

```
## AI Review Report

### 变更概要
- 变更文件数: N
- 涉及分层: Controller / Service / Repository / DTO / 其他
- 变更类型: 新增功能 / 修改功能 / 修复 / 重构

### 2.1 过度设计
- 结论: ✅ 通过 / ⚠️ 存在风险 / ❌ 不通过
- 详情: (如有问题，逐项列出；通过则写"未发现过度设计")

### 2.2 误解需求
- 结论: ✅ 通过 / ⚠️ 存在风险 / ❌ 不通过
- 详情: (如有问题，逐项列出；通过则写"实现与契约/fixture一致")

### 2.3 测试自嗨
- 结论: ✅ 通过 / ⚠️ 存在风险 / ❌ 不通过
- 详情: (如有问题，逐项列出；通过则写"测试有效验证业务行为")

### 总结
- 总体结论: ✅ 可提交人工 Review / ⚠️ 需关注后提交 / ❌ 需修复后重新审查
- 必须修复项: (列出 ❌ 项，无则写"无")
- 建议关注项: (列出 ⚠️ 项，无则写"无")
```

## 规则

- 审查必须基于实际 diff 内容，不得凭空猜测
- 每个检查项都必须有明确结论，不得含糊带过
- 发现问题时，必须指出具体代码位置和改进建议
- ❌ 项为必须修复，修复后需重新运行 agent-check.sh + 本 skill
- ⚠️ 项为建议关注，由人工 Review 做最终判断
